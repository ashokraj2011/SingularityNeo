SingularityNeo: Enterprise Architecture Deep-Dive

This document provides a detailed breakdown of the two core subsystems powering the SingularityNeo platform: the Token Optimization Framework and the Agent Learning Model.

Part 1: Token Optimization Framework

The Token Optimization strategy attacks token waste across all three phases of an LLM generation: Input (what the agent sees), Execution (how long sessions are remembered), and Output (what the agent types back).

Strategy 1: Input Truncation

Goal: Stop unnecessary tokens from ever entering the System Prompt.

  • 1. Phase-Sliced Guidance: Agents were historically loaded with massive instruction files. Now, SingularityNeo filters the injected rules based on the active SDLC phase. A BUILD phase developer agent never reads GOVERNANCE policies.
  • 2. Semantic Hunk Reads (workspace_read): Using the native TypeScript AST compiler, the backend extracts the sliceStartLine to sliceEndLine and injects only that function hunk into the prompt. Saves 80–95% of input tokens.
  • 3. The Context Budgeter: A mathematically precise Eviction Engine drops items based on Priority when limits are hit: SYSTEM_CORE > PHASE_GUIDANCE > WORK_ITEM_BRIEFING > RECENT_CHAT_HISTORY > CODE_HUNKS > MEMORY_HITS.

Strategy 2: Execution Compression

Goal: Prevent long-running autonomous tasks from expanding the context window to infinity.

  • 4. Tool-Loop History Rollup: SingularityNeo intercepts localMessages. When the array hits ~10 turns, it passes the oldest turns to a cheap model (gpt-4o-mini) to compress the history.
  • 5. Structured Rollup Output: The budget model outputs a highly compressed JSON state note (currentGoal, lastAction, blocker). This is injected into the prompt, keeping state awareness while only paying for 6 raw turns.
  • 6. Multi-Symbol Retrieval Bundles: Agents can pass includeCallers and includeCallees to workspace_read, retrieving the target function and dependencies in a single, cheap payload.

Strategy 3: Output Constraints

Goal: Force the LLM to type as few characters as mathematically possible.

  • 7. Diff-First Prompting: System prompts aggressively steer the agent towards using workspace_apply_patch or workspace_replace_block.
  • 8. Diff Enforcement Policy: If an agent calls workspace_write on an existing file twice, it returns a recoverable error, pushing the agent toward producing unified diffs.
  • 9. Prompt Receipts: The platform tracks exactly which fragments were included, evicted, and how many tokens were burned per model call.

Part 2: The Agent Learning Model

Unlike standard coding assistants, the SingularityNeo Learning Model is built as a Continuous Intelligence Engine. It studies past executions, learns from mistakes, and instantly absorbs human feedback.

  • 1. Four-Tiered Memory Hierarchy:
    • HOT: Active Session uncompressed logs.
    • WARM: Recent Work Items (rank highly in vectors).
    • COLD: Historical bugs (isolated).
    • ARCHIVE: Aged-out records.
  • 2. The Distillation Engine: Passes the completed session to a background model which extracts the pure signal ("What was the goal? What error occurred?"). It overrides bad future paths.
  • 3. Graceful Vector Storage: If Postgres pgvector is available, ANN searches are executed natively. Otherwise, it gracefully degrades to local JSON-Cosine similarity.
  • 4. Human Operations Contract: When an Operator corrects an agent, it is pushed into an Operations Contract injected at the top of the System Prompt, structurally superseding AI hallucination loops.
  • 5. Incremental & Symbol-Deep Indexing: The ingest engine hashes file checkpoints. Only changed files are embedded. The Directed Edge Graph maps dependencies (e.g. `login` CALLS `validate`).
